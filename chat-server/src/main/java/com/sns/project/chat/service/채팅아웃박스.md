# 채팅 Outbox 설계 정리

## 목표

- `ChatMessage` 저장과 Kafka 이벤트 발행을 느슨하게 연결한다.
- DB 저장은 성공했는데 Kafka 발행이 실패해서 `broadcast`, `unread`, `notification` 처리가 빠지는 문제를 막는다.
- 현재의 서버 내부 `AFTER_COMMIT` 이벤트 구조를, 나중에 `Outbox -> Kafka -> Consumer` 구조로 자연스럽게 옮긴다.

## 현재 구조

지금은 대략 아래 순서다.

1. 메시지 저장
2. `ChatRoom.latestMessageId/latestMessageAt` 갱신
3. `AFTER_COMMIT` 내부 이벤트 발행
4. Redis unread projection 갱신

이 구조는 `트랜잭션 커밋 후에만 Redis를 건드린다`는 점은 좋다.
하지만 이벤트 전달이 아직 JVM 내부에 묶여 있어서, 나중에 서버가 여러 대가 되거나 Kafka 비동기 consumer로 분리할 때 한계가 있다.

## 최종 목표 구조

1. 메시지 전송 요청
2. DB 트랜잭션 안에서
   - `ChatMessage` 저장
   - `ChatRoom.latestMessageId/latestMessageAt` 갱신
   - `OutboxEvent` 저장
3. 트랜잭션 커밋
4. `OutboxRelay`가 `PENDING` 상태 이벤트를 읽는다
5. Kafka 토픽으로 발행한다
6. 발행 성공 시 `OutboxEvent`를 `PUBLISHED`로 바꾼다
7. Kafka consumer group이 각자 처리한다
   - `broadcast`
   - `unread projection`
   - `notification`

## 왜 Outbox가 필요한가

Outbox가 없으면 아래 상황이 생길 수 있다.

1. DB 메시지 저장 성공
2. Kafka 발행 실패

이 경우 메시지는 DB에 존재하지만:

- 상대방에게 실시간 전파되지 않을 수 있고
- unread count가 올라가지 않을 수 있고
- 알림도 발행되지 않을 수 있다

즉 `원본 데이터`와 `비동기 파생 처리`가 서로 어긋난다.

Outbox는 이 문제를 줄이기 위해:

- 같은 DB 트랜잭션 안에서
- "나중에 반드시 발행해야 할 이벤트"를 같이 저장한다.

## Outbox 테이블 설계

엔티티 이름 추천:

- `OutboxEvent`

컬럼 추천:

- `id`
- `aggregateType`
  - 예: `CHAT_MESSAGE`
  - 예: `CHAT_ROOM_READ`
- `aggregateId`
  - 예: `chatMessage.id`
  - 읽음 이벤트면 `roomId`나 별도 의미 있는 aggregate id
- `eventType`
  - 예: `CHAT_MESSAGE_CREATED`
  - 예: `CHAT_ROOM_READ`
- `topic`
  - 예: `chat.message.created`
  - 예: `chat.room.read`
- `eventKey`
  - Kafka key
  - 보통 `roomId`
- `payload`
  - JSON 문자열
- `status`
  - `PENDING`
  - `PUBLISHED`
  - `FAILED`
- `retryCount`
- `publishedAt`
- `createdAt`

## 상태 enum

추천 enum:

- `OutboxStatus`

값:

- `PENDING`
- `PUBLISHED`
- `FAILED`

## 이벤트 타입

초기에는 두 개만 있으면 된다.

### 1. `CHAT_MESSAGE_CREATED`

payload 예시:

```json
{
  "roomId": 1,
  "messageId": 120,
  "senderId": 2,
  "content": "안녕하세요",
  "receivedAt": 1775017994
}
```

용도:

- STOMP broadcast
- Redis unread projection 증가
- notification

### 2. `CHAT_ROOM_READ`

payload 예시:

```json
{
  "roomId": 1,
  "userId": 3,
  "messageId": 120
}
```

용도:

- Redis unread projection 초기화

## 토픽 설계

권장 토픽:

- `chat.message.created`
- `chat.room.read`

이유:

- 메시지 생성과 읽음 처리는 의미가 다르다
- consumer group을 역할별로 나누기 쉽다

## Consumer Group 설계

### `chat-broadcast`

- consume: `chat.message.created`
- 역할: STOMP `/topic/chat/rooms/{roomId}` 로 전파

### `chat-unread`

- consume: `chat.message.created`
- 역할: Redis unread `+1`
- 단, sender 제외
- 단, 현재 room presence 있으면 증가하지 않음

- consume: `chat.room.read`
- 역할: Redis unread `0` 또는 해당 room field 삭제

### `chat-notification`

- consume: `chat.message.created`
- 역할: 알림 발송

## DB와 Redis 역할 분리

### DB

정합성 원본:

- `ChatMessage`
- `ChatRoom.latestMessageId`
- `ChatRoom.latestMessageAt`
- `ChatReadStatus.lastReadMessageId`
- `OutboxEvent`

### Redis

빠른 projection:

- `chat:unread:user:{userId}` hash
  - field: `roomId`
  - value: unreadCount

- `chat:presence:room:{roomId}:user:{userId}`
  - value: 접속 카운트

즉 unread count는 DB 원본이 아니라 Redis projection으로 둔다.

## Presence 설계

왜 필요한가:

- 사용자가 이미 방 안에서 메시지를 보고 있으면 unread를 올리면 안 된다.

권장 Redis key:

- `chat:presence:room:{roomId}:user:{userId}`

값:

- 소켓/구독 개수 카운트

동작:

- STOMP `SUBSCRIBE`: `+1`
- STOMP `DISCONNECT`: `-1` 또는 0이면 삭제
- unread 증가 전 presence 확인

## 서비스 레벨 구현 순서

### 메시지 저장

현재:

- `ChatService.saveMessage()`

최종 목표:

1. `ChatMessage` 저장
2. `ChatRoom.updateLatestMessage()`
3. `OutboxEvent(CHAT_MESSAGE_CREATED)` 저장

위 3개를 같은 트랜잭션 안에서 처리

### 읽음 처리

현재:

- `ChatRoomService.markRoomAsRead()`

최종 목표:

1. `ChatReadStatus.lastReadMessageId` 갱신
2. `OutboxEvent(CHAT_ROOM_READ)` 저장

이 둘을 같은 트랜잭션 안에서 처리

## Outbox Relay 설계

클래스 예시:

- `OutboxRelayService`
- `OutboxRelayScheduler`

동작:

1. `status = PENDING` 이벤트를 배치로 조회
2. 오래된 순서로 Kafka 발행
3. 성공 시 `PUBLISHED`
4. 실패 시 `retryCount + 1`
5. 일정 횟수 초과 시 `FAILED`

조회 예시:

- `findTop100ByStatusOrderByIdAsc(PENDING)`

주의:

- 처음엔 단일 인스턴스 기준으로 시작해도 됨
- 나중에 여러 인스턴스로 가면
  - 상태 선점
  - `FOR UPDATE SKIP LOCKED`
  - 배치 락
  같은 보강이 필요함

## 현재 코드와의 연결점

현재 임시 구조:

- `ChatRealtimeProjectionEventHandler`
- 서버 내부 `AFTER_COMMIT` 이벤트 기반

이건 중간 단계로 유지 가능하다.
최종적으로는:

- 내부 이벤트 발행 대신 `OutboxEvent` 저장
- 내부 핸들러 대신 Kafka consumer 처리

즉 나중에 제거되거나 축소될 대상이다.

반면 아래는 재사용 가치가 높다.

- `ChatRealtimeStateService`

이 서비스는 Redis projection 로직을 담고 있으므로,
나중에 Kafka consumer에서도 그대로 호출하면 된다.

## 최종 구현 순서 추천

1. `OutboxEvent` 엔티티/리포지토리 추가
2. `OutboxStatus` enum 추가
3. 메시지 저장 시 outbox 저장
4. 읽음 처리 시 outbox 저장
5. `OutboxRelayScheduler` 추가
6. Kafka 원본 이벤트 토픽 추가
7. `broadcast` consumer를 `chat.message.created` 기반으로 전환
8. `unread projection`을 Kafka consumer로 전환
9. 기존 내부 `AFTER_COMMIT` 핸들러 제거

## 한 줄 결론

최종 목표는 아래다.

- `DB 트랜잭션 안에 원본 데이터 + Outbox 저장`
- `Outbox Relay가 Kafka 발행`
- `broadcast / unread / notification`은 consumer group이 각자 처리

이 구조로 가면 채팅 서비스가 커져도 정합성과 확장성을 함께 가져갈 수 있다.
